Memory System

Give your agents durable, queryable memory that survives restarts, syncs across daemons, and never exposes raw data to the cloud.

Overview

Every agent running on a Synapse daemon gets its own isolated, persistent memory store. By default the store is a local SQLite database; opt in to Chroma for semantic vector search on top.

Memory survives daemon restarts and process crashes. After each run completes the daemon automatically syncs the agent's memory to the Synapse cloud as end-to-end-encrypted blobs — the same recovery key used for checkpoint encryption protects memory blobs. Any other authorized daemon can restore the snapshot, making memory portable across machines.

Before any data leaves the daemon, the redaction pipeline (shared with run telemetry) strips PII and secrets. The cloud stores only opaque ciphertext; it never executes agents or holds raw key material.

  • Storage: SQLite KV store (+ optional Chroma vector index) on the daemon
  • Sync: automatic after each run; manual via CLI
  • Encryption: E2E to org recovery key before leaving the daemon
  • Redaction: PII/secret scrubbing before every sync
  • Quota: 50 MB KV + 500 MB vector per agent (configurable)

Key-Value Store

The KV store is the primary memory API. Each entry is a structured record with the following fields:

Field Type Description
namespace string Logical grouping for entries. Examples: context, user-prefs, session. Namespaces are created on first write.
key string Identifier within the namespace. Must be unique per namespace.
value any Any JSON-serializable value: string, number, boolean, object, or array.
tags string[] Optional list of tag strings for filtering and search. Tags are indexed.
ttl duration | null Optional time-to-live. Entry is automatically deleted after this duration. Accepts ISO 8601 durations (e.g. PT1H) or seconds integer.
size_bytes int Byte size of the serialized value. Tracked automatically for quota enforcement; read-only.

Memory API

Agents interact with memory through the memory object injected into every run context. Each call is an MCP tool invocation under the hood, so it is auditable in the memory journal and visible in the Web UI.

Set a value

# Set a value
memory.set("context", "last-pr-reviewed", {"number": 42, "status": "approved"}, tags=["pr"])

Get a value

# Get a value
val = memory.get("context", "last-pr-reviewed")

Search by tag

# Search by tag
results = memory.search(tags=["pr"], namespace="context")

Delete an entry

# Delete
memory.delete("context", "last-pr-reviewed")

List a namespace

# List namespace
entries = memory.list("context", limit=50)

All five operations are synchronous from the agent's perspective. The daemon handles SQLite I/O and journal writes in a background thread without blocking the agent run.

Memory Journal

Every memory operation is appended to an append-only write-ahead log stored in SQLite alongside the KV store. The journal is the authoritative record of all mutations and is used during cloud restore to resolve conflicts when two daemons have diverged.

Each journal entry contains:

Field Description
operation set or delete
namespace Namespace of the affected entry
key Key of the affected entry
timestamp ISO 8601 UTC timestamp of the operation
run_id ID of the agent run that triggered the operation; null for manual edits via Web UI or CLI

The journal is visible in the Web UI under Agent → Memory → Journal tab. Entries are immutable; the journal grows indefinitely by design. Old entries are compacted (not deleted) when a cloud sync checkpoint is created.

Cloud Sync

Memory sync happens automatically after each agent run completes. The daemon packages the current KV snapshot plus the journal delta since the last sync, encrypts the bundle with the org recovery key (the same key used for run checkpoint encryption), and uploads the opaque blob to the Synapse cloud.

The cloud stores the blob but cannot read it — it is a transparent relay. Any daemon that holds the org recovery key can pull and decrypt the blob to restore an agent's memory on a new machine.

Sync status is visible in the Web UI under Agent → Memory → Sync tab, showing the last sync time, blob size, and any pending local changes.

To trigger a manual sync:

terminal
$ synapse agent memory sync pr-reviewer
Syncing memory for agent "pr-reviewer"...
KV snapshot: 128 entries (42 KB)
Journal delta: 17 new entries
Encrypted and uploaded in 340 ms

Redaction Before Sync

The same PII and secret redaction pipeline used for run telemetry runs on all memory values before they leave the daemon. Redaction applies to both KV values and the write-ahead journal entries.

ℹ

Redacted tokens appear as <REDACTED:TYPE:hash> in cloud blobs — for example, <REDACTED:EMAIL:a3f2c1> or <REDACTED:API_KEY:9e7d04>. The daemon stores a local mapping from hash to original value so the agent can transparently reconstruct the value on read. The cloud never sees raw memory contents.

Redaction covers:

  • Email addresses, phone numbers, and other PII patterns
  • API keys, tokens, and secrets matching the daemon's secret-detection rules
  • Any value that matches a custom redaction pattern configured in daemon.toml

Redaction is one-way for the cloud: even if someone obtained the encrypted blob and the recovery key, they would see only hashed placeholders for sensitive tokens rather than raw values.

Viewing and Editing Memory in Web UI

Navigate to Web UI → Agent → Memory to inspect and manage an agent's memory store without writing code or using the CLI.

The Memory tab has three sub-tabs:

Browser tab

  • Left sidebar — namespace browser listing all namespaces with entry counts and total size
  • Main area — key-value table for the selected namespace; columns: key, value preview, tags, TTL, size, last modified
  • Tag filter — filter entries by one or more tags
  • Key prefix search — live search by key prefix within the selected namespace
  • JSON editor — click any row to open the full value in a syntax-highlighted editor; save changes directly
  • Delete — delete individual entries, bulk-delete by namespace, or bulk-delete by tag
  • Quota bar — shows current KV usage vs the agent's quota limit

Journal tab

Append-only log of all memory operations with operation type, key, timestamp, and run ID. Filterable by operation type, namespace, and date range.

Sync tab

Shows last sync timestamp, encrypted blob size, pending local changes since last sync, and a Sync now button.

Pre-loading Memory Snapshots

Admins can export a full memory snapshot to JSON and import it into any agent — useful for migrating an agent between organizations, seeding context before a first run, or creating a checkpoint before a major change.

Export

terminal
# Export all namespaces to a JSON file
$ synapse agent memory export pr-reviewer > memory.json
Exported 3 namespaces, 247 entries (89 KB) to memory.json

Import

terminal
# Import a snapshot into a new agent
$ synapse agent memory import pr-reviewer-v2 memory.json
Importing 3 namespaces, 247 entries...
Import complete. Memory synced to cloud.

The import command merges the snapshot with any existing memory using last-write-wins semantics. Existing entries not present in the snapshot are preserved. Pass --replace to wipe all existing memory before importing.

Vector Backend (Chroma)

The optional vector backend enables semantic (similarity) search over memory values. It is implemented with Chroma DB running locally on the daemon.

Installation

pip install "synapse-worker[vector]"

Configuration

Add the following to your daemon.toml:

[memory]
vector_backend = "chroma"
chroma_path    = "~/.synapse/chroma"

Semantic search

results = memory.semantic_search(
    "what PRs did I review last week?",
    namespace="context",
    top_k=5
)

Semantic search returns entries ranked by cosine similarity to the query embedding. Each result includes the entry's key, value, tags, and a score float between 0 and 1.

The Chroma index is included in cloud sync alongside the KV snapshot — both are encrypted to the org recovery key before upload. Restoring memory on a new daemon restores the vector index as well, so semantic search is available immediately after restore without re-indexing.

Quotas

Each agent has an independent memory quota enforced by the daemon. Default limits:

Store Default limit
KV store (size_bytes sum) 50 MB
Vector index (Chroma) 500 MB

When an agent exceeds its quota, new memory.set() calls raise a MemoryQuotaError. Existing entries are not affected and reads continue to work normally.

Quota limits are configurable per agent in Web UI → Agent → Settings → Memory. The current usage vs limit is shown as a quota bar in the Memory Browser tab.

✓

Use ttl on entries that are only relevant for a limited window — for example, session context or per-run state. Automatic expiry keeps quota usage low without requiring explicit cleanup logic in your agent code.
