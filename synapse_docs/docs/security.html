SECURITY

Security Architecture

How Synapse keeps your secrets safe, redacts sensitive data on-device, and enforces a hard trust boundary between cloud and execution.

The Trust Model

Synapse is built around a hard trust boundary: the cloud is an authenticated relay and coordination layer, never an execution layer. Your secrets, your agent output, and your private keys stay on your machines. The cloud stores what it needs to route and coordinate, and nothing more.

The cloud SEES The cloud NEVER SEES
Agent names and configs Raw API keys
Variable names (not values) Env var values (ciphertext only)
Redacted telemetry / traces Raw telemetry before redaction
Daemon hostname, OS, IP Private encryption keys
Run status and metadata Execution subprocess output
Encrypted checkpoint blobs Checkpoint contents
This design means a full cloud compromise would not expose your API keys, env var values, or agent execution data. The cloud stores only encrypted ciphertext it cannot read — and it never holds the private key needed to decrypt it.

Device Authorization (OAuth 2.0 RFC 8628)

Synapse uses the OAuth 2.0 Device Authorization Grant (RFC 8628) to authenticate daemons. This means no passwords in the terminal — no typing credentials into a shell, no SSH tunnels, no pasting tokens into config files.

The flow authenticates through a browser session the user has already established, with an out-of-band confirmation step so the user can verify they are authorizing the right machine.

Authorization Flow

  1. 1
    Request device code

    The daemon calls POST /auth/device/code and receives a short-lived user_code (e.g., SYNX-4821), a verification_uri, and a poll interval.

  2. 2
    Display code to user

    The CLI prints the human-readable code alongside a QR code and instructs the user to open the verification URI in their browser.

  3. 3
    User reviews device metadata

    The Web UI shows the pending authorization request with full device context: hostname, OS, platform, source IP, approximate city, and when the request was made. The user can confirm these match the machine they intend to authorize.

  4. 4
    Approve or Deny

    The user clicks Approve to authorize the device, or Deny to reject it. A deny invalidates the device code immediately.

  5. 5
    Daemon polls for token

    The daemon polls /auth/device/token at the specified interval. Responses are authorization_pending until the user acts, then resolve to a token pair on approval or an access_denied error on denial.

  6. 6
    Receive token pair

    On approval, the daemon receives a short-lived access token (~15 min TTL) and a per-device rotating refresh token.

  7. 7
    Tokens stored in OS keychain

    Tokens are written to the OS keychain (macOS Keychain, Windows Credential Manager, or libsecret on Linux) — not to disk in plaintext. Each token refresh returns a new refresh token, rotating credentials continuously.

› Requesting device code... ✓

› Authorization URL: https://app.synapse.ai/device

› Your code: SYNX-4821

› Waiting for approval in browser...

› Device approved by user@example.com ✓

Token Properties

Property Detail
Access token TTL ~15 minutes
Refresh tokens Per-device, rotating — each use returns a new token
Token storage OS keychain (never plaintext on disk)
Revocation One-click in Web UI; invalidates all tokens for that daemon instantly
Blast radius Revoking one device does not affect other daemons or the user's password

Encryption Keys (X25519 Keypair)

On first login, each daemon generates a unique X25519 keypair. This keypair is the root of the end-to-end encryption system for environment variables and agent checkpoints.

  • Private key: stored exclusively in the OS keychain. Never transmitted to the cloud. If no keychain is available (headless servers), the private key is written to an encrypted file with 0600 permissions.
  • Public key: registered with the cloud during device pairing. Used by the Web UI to encrypt env var values. The encrypted ciphertext is what flows through the cloud — the cloud never holds the private key needed to decrypt it.

OS Keychain Backends

Platform Backend
macOS macOS Keychain
Windows Windows Credential Manager
Linux (with GUI) GNOME Keyring / libsecret
Linux (headless) Encrypted file with 0600 permissions

End-to-End Environment Variable Encryption

Environment variable values are encrypted in the browser before they ever leave your device, using the target daemon's X25519 public key. The cloud receives and stores ciphertext — only the daemon that holds the matching private key can decrypt it.

Encryption Flow

  1. 1
    User enters a secret value

    The user types (or pastes) a secret into the Environment tab in the Web UI. The value exists only in browser memory at this point.

  2. 2
    Browser fetches the daemon's public key

    The Web UI retrieves the daemon's registered X25519 public key from the cloud. This key was registered during device pairing and is safe to transmit.

  3. 3
    Browser encrypts using sealed-box encryption

    The value is encrypted with sealed-box encryption: X25519 key agreement + XSalsa20-Poly1305 authenticated encryption, implemented via libsodium (via the tweetnacl JS library). The resulting ciphertext is authenticated — tampering is detectable.

  4. 4
    Ciphertext sent to cloud

    The browser sends only the variable name (plaintext) and the encrypted value (ciphertext) to the cloud. The cloud stores both. The plaintext value is never sent over the network.

  5. 5
    Cloud relays ciphertext to daemon

    When the daemon needs the variable (e.g., at agent run start), the cloud relays the ciphertext. The cloud cannot decrypt it — it does not hold the private key.

  6. 6
    Daemon decrypts with its private key

    The daemon retrieves its private key from the OS keychain and uses it to decrypt the sealed box. This operation happens entirely on the daemon's machine.

  7. 7
    Plaintext injected into subprocess environment

    The decrypted value is placed directly into the agent subprocess's environment variables. It is never written to disk and never logged.

Data Flow Diagram

Browser                     Cloud                    Daemon
  │                           │                        │
  │ encrypt(value, pk_daemon) │                        │
  │──────────────────────────►│                        │
  │                           │   relay(ciphertext)    │
  │                           │───────────────────────►│
  │                           │                        │ decrypt(ciphertext, sk_daemon)
  │                           │                        │ → plaintext → subprocess env
Even if the cloud is fully compromised, the attacker has only ciphertext they cannot decrypt. The daemon's private key never leaves the machine — it is stored in the OS keychain and never transmitted.

On-Device Redaction

Before any telemetry leaves the daemon, it passes through a two-layer streaming redaction pipeline. Both layers run on the daemon — the cloud never receives unredacted output.

Layer A — Regex + Entropy-Based Detection

Layer A is always active, adds ~zero latency, and catches the most common categories of secrets:

  • API key patterns: regex patterns for common key formats — OpenAI (sk-...), Anthropic (sk-ant-...), AWS access keys, GitHub personal access tokens, Stripe secret keys, and dozens more.
  • High-entropy strings: substrings with entropy above a configurable threshold are flagged as potential secrets, regardless of format.
  • Replacement format: redacted values are replaced with a salted token like <REDACTED:API_KEY:a91f>. The salt is derived from the original value — identical secrets get identical tokens, enabling deduplication tracking without re-exposing the secret.
Redaction — Layer A

Input: Using key sk-ant-api03-XXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXX

Output: Using key <REDACTED:API_KEY:a91f>

Layer B — Presidio PII Guard (Optional)

Layer B adds a full PII detection pipeline using Microsoft Presidio, running in a CPU-bound ProcessPoolExecutor to avoid blocking the asyncio event loop.

Entity types detected:

  • Full names and partial names
  • Email addresses
  • Phone numbers (international formats)
  • Social Security Numbers (SSNs)
  • Credit card numbers (major card patterns)
  • IP addresses and URLs
  • Prompt injection / jailbreak attempts

Install with:

Terminal

pip install synapse-worker[presidio]

Layer B adds approximately 100–200 ms of latency per chunk, depending on chunk size and hardware. It requires downloading NLP models on first run.

Even with Presidio disabled, Layer A is always active and will catch most secrets. Enable Layer B (Presidio) when dealing with user-submitted content that may contain PII such as names, phone numbers, or identification numbers.

Ruleset Engine (On-Device Guardrails)

The Ruleset Engine enforces policies on-device before agent actions are executed. Every tool call, shell command, file write, and HTTP request is validated against the active ruleset. Enforcement is deterministic — it does not rely on prompting the model.

Ruleset Types

Type What it controls
command blockers Shell commands matching a glob or regex pattern
write_path guards Directory paths the agent is permitted to write to
network allow-list Domains and IP ranges the agent's HTTP calls may reach
cost caps Maximum USD spend threshold per run
tool_call caps Maximum number of tool invocations per run
capability gating Which MCP capabilities and tools the agent may use

Severity Levels

Each rule carries a severity level that determines what happens on a match:

  • block The action is unconditionally denied. The agent receives a refusal response; the run continues unless the agent gives up.
  • require-approval The run pauses and sends a Human-in-the-Loop (HITL) notification. An authorized operator must approve or deny the specific action before the agent proceeds.
  • warn The action is permitted but logged as a warning in the run trace. Useful for observing boundary-approaching behavior without blocking.
Guardrails are enforced by the daemon binary, not by prompting the model to follow rules. The model's output is checked against the ruleset after generation — the model cannot "jailbreak" the guardrails by producing different text.

Org Recovery Keys

Long-running agents produce checkpoints that can be used to resume a run after interruption. Synapse encrypts these checkpoints so they can be stored in the cloud without exposing their contents.

  1. 1
    Org keypair generated on creation

    When an org is created, an org-scoped X25519 keypair is generated. The private half is distributed only to authorized daemon keychains; the public half is registered with the cloud.

  2. 2
    Checkpoints encrypted to org public key

    Before a run checkpoint is sent to the cloud for storage, the daemon encrypts it using the org's public recovery key. The cloud stores only the encrypted blob.

  3. 3
    Any authorized daemon can resume

    If a machine is lost or a daemon is replaced, any authorized daemon in the org can request the encrypted checkpoint from the cloud. It decrypts the blob using the org private key from its keychain and resumes the run from the last checkpoint.

Total hardware loss does not mean total run loss. As long as the org's private key exists in at least one authorized daemon's keychain, all checkpoints can be recovered and runs resumed.

RBAC (Role-Based Access Control)

Synapse uses a four-tier role hierarchy. Roles are assigned per-member per-org and are enforced at every layer of the stack — API middleware, database policies, realtime subscriptions, and daemon command authorization.

Role Hierarchy

Role Key Permissions
owner Billing, all settings, transfer/delete org, all admin/operator/viewer actions
admin Deploy and edit agents, manage members, approve HITL gates, view secret names
operator Deploy and edit agents, approve HITL gates — no access to secret values
viewer Read-only access to runs, traces, alerts, and dashboards

Enforcement Layers

RBAC is applied at four independent layers — a misconfiguration in one layer does not expose data through another:

  • REST API: FastAPI authentication middleware resolves the JWT on every request, extracts org_id and role, and rejects operations that exceed the caller's role before any database query runs.
  • Database: Postgres Row-Level Security (RLS) policies on every table enforce org-scoping and role-based access. Even if the API layer were bypassed, direct database queries would return only the rows the JWT's role permits.
  • Realtime: Supabase Realtime channels are gated by RLS. Subscriptions that would return unauthorized rows are rejected at the channel level — tenants cannot observe each other's events.
  • Daemon: Commands from the cloud carry signed authorization tokens. The daemon verifies the signature and the embedded role claim before executing any command — a compromised relay cannot issue unauthorized commands.

Multi-Factor Authentication (MFA)

Synapse supports two MFA methods. Both can be enrolled simultaneously; at login the user can choose either.

TOTP (Time-Based OTP)

Works with any standards-compliant TOTP authenticator: Google Authenticator, Authy, 1Password, Bitwarden, and others.

  • Setup: Account Security → Add authenticator app → scan the QR code → enter a verification code to confirm enrollment.
  • Recovery codes: Provided at enrollment. Store them securely — they are the only way to regain access if you lose your authenticator device.

WebAuthn / Passkeys

Supports hardware security keys (YubiKey, etc.) and platform authenticators (Touch ID, Face ID, Windows Hello).

  • Setup: Account Security → Add security key → the browser prompts for credential creation via the WebAuthn API.
  • Phishing-resistant: WebAuthn credentials are cryptographically bound to the origin (app.synapse.ai). A credential registered at Synapse cannot be used on a phishing site.

Org-Level MFA Policy

Org owners can require MFA for all members from Settings → Authentication. When the policy is active:

  • Members without MFA enrolled cannot log in.
  • Members are prompted to enroll before their next login is permitted.
  • Per-member MFA enrollment status is visible to admins in the Members list.
Enabling org-level MFA enforcement is strongly recommended for production orgs. It ensures that a compromised password alone is insufficient to access your agents.

Command Signing (ed25519)

Every command sent from the cloud to a daemon is signed with an ed25519 key. The daemon verifies the signature before executing any command. This ensures that even if the cloud is compromised or the communication channel is intercepted, an attacker cannot forge or replay commands.

Signed Command Fields

Each signed command includes:

  • verb — the action being requested (e.g., run_agent, stop_run)
  • agent_id — the target agent
  • daemon_id — the target daemon
  • org_id — the issuing organization
  • user_id — the user who authorized the command
  • payload — the command-specific data
  • nonce — a unique random value preventing replay
  • timestamp — issuance time for freshness verification

Verification and Rejection

When the daemon receives a command, it:

  1. Verifies the ed25519 signature using the session signing public key.
  2. Checks the nonce against a short-lived seen-nonces store — rejects duplicates.
  3. Validates the timestamp is within an acceptable skew window.
  4. Confirms the daemon_id and org_id match its own identity.

Commands that fail any check are rejected and logged as security events, which trigger an alert visible to org admins.

Command signing is enabled automatically for all daemon sessions established after v0.8. Daemons running older versions fall back to unsigned commands. Upgrade to v0.8+ to enable this protection.

Row-Level Security (Postgres RLS)

All data in Synapse's Supabase backend is protected by Postgres Row-Level Security (RLS) policies. RLS is the final safety net: even a bug in application-layer authorization cannot return data that the policy forbids.

Policy Design

  • Every table has RLS enabled. There is no table in the Synapse schema without an active RLS policy. The default posture is deny-all; policies grant access, not restrict it.
  • Org-scoping: All policies include an org_id = auth.jwt() ->> 'org_id' predicate. A query from org A cannot return rows belonging to org B — even if the application code contains a bug that omits a WHERE clause.
  • Role-based access: Policies on sensitive tables (e.g., environment variable names, audit logs) additionally check the role claim in the JWT. A viewer-role JWT cannot select secret names even via a direct Supabase client.
  • JWT carries claims: The user's JWT, issued by Supabase Auth, contains org_id, role, and user_id. Supabase validates the JWT signature before any row is returned — the claims cannot be forged without the signing secret.
  • Service-role key: The backend's service-role key bypasses RLS (by design, for server-side operations). It is never exposed to browsers, never included in client-side bundles, and rotated on suspicion of compromise.
Browser queries use the anon key + user JWT and are fully subject to RLS. Server-side queries use the service-role key only when cross-tenant operations are legitimately required (e.g., billing webhooks) — all such operations are explicitly audited in the codebase.
