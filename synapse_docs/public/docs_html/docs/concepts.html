CORE CONCEPTS

How Synapse works

Synapse is built around a strict control-plane / data-plane split. Understanding the key concepts helps you use the platform safely and effectively.

The Control Plane / Data Plane Split

Synapse enforces a hard trust boundary between what runs in the cloud and what runs on your machine. This is not a preference — it is an architectural invariant that cannot be bypassed.

Control Plane (cloud + browser)          Data Plane (your machine)
─────────────────────────────────────    ──────────────────────────────
• routes commands                        • executes agents
• stores audit trail                     • holds raw API keys
• delivers telemetry to browser          • redacts PII before sending
• manages RBAC & auth                    • enforces guardrails
• provides realtime UI updates           • checkpoints long runs

The cloud is a broker and historian, not an executor. Everything sensitive — execution, secrets, redaction — happens on the daemon. Browser and daemon never talk directly; the cloud only relays encrypted commands.

Three Invariants:
  1. Browser and daemon never talk directly.
  2. Cloud never executes agents or holds raw provider keys.
  3. Daemon connects outbound-only — no inbound ports required.

Daemons

A daemon is an instance of synapse-worker running on one of your machines. The daemon is the data plane: it holds secrets, runs agents, and enforces guardrails locally.

Each daemon:

  • Has a unique identity — an X25519 keypair generated at first login
  • Connects to the cloud via an outbound-only WebSocket uplink
  • Stores secrets in the OS keychain (Keychain / Credential Manager / Secret Service)
  • Executes agents as isolated subprocesses
  • Enforces guardrails before and after each agent action
  • Reports heartbeats so the cloud knows it is online

Daemon lifecycle

install → login → init → run → heartbeat loop
   │         │       │      │
   │         │       │      └─ emit heartbeat every 30 s
   │         │       └─ generate keypair, register with cloud
   │         └─ authenticate org membership, store token
   └─ pip install synapse-worker  (or brew / winget)

Agents

An agent is a configured task executor. Agents come in two types, both running on a daemon and governed by the same ruleset/guardrail system.

API models LLM-based

API model agents call an LLM endpoint (Claude, GPT-4, Gemini, or any OpenAI-compatible API):

  • Run a prompt through an LLM API
  • Can use tools and capabilities (MCP servers, plugins)
  • Support streaming reasoning and multi-turn conversations
  • Cost tracking: token usage mapped to USD spend

CLI tools Shell-based

CLI tool agents spawn a subprocess (Claude Code, Codex CLI, aider, or any shell command):

  • Spawn a subprocess with the configured command and arguments
  • Stream stdout/stderr directly to the trace viewer
  • Token and cost accounting via output parsing
  • Platform-scoped: configure different commands for Windows / macOS / Linux

Agent properties

Runs

A run is one execution of an agent. Every run is fully traced and stored as part of the audit trail.

Trigger types

TriggerHow it fires
manualInitiated from the Web UI or TUI by a user
scheduleFires on a cron expression configured on the agent
webhookTriggered by an inbound HTTP call to the cloud endpoint
agentSpawned by another running agent (orchestration / lineage)

Run status flow

           ┌─────────┐
  trigger ─▶ running │
           └────┬────┘
                │
        ┌───────┴────────┐
        ▼                ▼
   ┌─────────┐     ┌──────────┐
   │ blocked │     │recovering│  ← daemon crashed mid-run
   │ (HITL)  │     └────┬─────┘
   └────┬────┘          │ daemon restarts

        ▼                ▼
   ┌──────┐         ┌──────┐
   │passed│         │passed│
   └──────┘         └──────┘
       (deny → aborted)

Run recovery: if the daemon crashes mid-run, the run is marked recovering and resumes from the last checkpoint when the daemon comes back online.

Trace contents

Each run produces a trace — a log of every model response, tool call, and system event. Traces are stored in the cloud (after on-device redaction) and viewable in the Web UI.

Lineage

Runs can have an optional parent run (when triggered by an agent). This forms a lineage tree — useful for tracing orchestrated multi-agent workflows back to the originating run.

Human-in-the-Loop (HITL) Gates

A HITL gate pauses a run and waits for a human decision before the agent can continue. This is the primary safety mechanism for high-risk actions.

When an agent action matches a ruleset entry with severity require-approval or block, the daemon pauses execution and routes an approval request to:

  • The Web UI approvals queue
  • Slack, Discord, or email channels (if configured)
  • The local TUI running on the daemon's machine
Who can approve? Only members with the admin or operator role can approve or deny HITL gates. The decision is audit-logged with a timestamp, approver identity, and rationale.

Approving sends a signed resume signal back to the daemon. Denying sends an abort signal. In both cases the decision is appended to the run's trace and is immutable.

Guardrails (Ruleset Engine)

Guardrails are policies enforced by the daemon — not the model. They run before the model's output is acted on, making them robust to prompt injection and jailbreak attempts.

Guardrails are enforced at the daemon level and cannot be bypassed by model output. A model that generates a blocked command causes the run to pause, not execute.
Guardrail typeWhat it does
Command blockers Deny-list of destructive commands (rm -rf, DROP TABLE, fork bombs, etc.)
Write-path guards Allow-list of directories the agent can write to
Network allow-list Domains the agent's HTTP calls can reach
Cost cap Hard stop when USD spend exceeds a configured threshold
Tool-call cap Hard stop when tool invocations exceed a configured count
Capability gating Which installed capabilities the agent is permitted to use

Ruleset entries specify a pattern (what to match), a severity (warn / require-approval / block), and an optional message shown to the approver.

Secrets & Encryption

Secrets never leave your machine in plaintext. The flow uses libsodium sealed-box encryption end-to-end:

  1. You type a secret value in the Web UI.
  2. The browser encrypts it with the daemon's X25519 public key (sealed-box, libsodium) before any network call.
  3. The encrypted ciphertext is sent to the cloud — the cloud stores the ciphertext and variable name only.
  4. The daemon decrypts with its private key, which is stored in the OS keychain and never leaves the machine.
  5. The plain value is injected into the agent's subprocess environment at runtime.
The cloud never sees the plain value of any secret. Even if the cloud database were compromised, an attacker would obtain only ciphertexts that require your daemon's private key to decrypt.

On-Device Redaction

Before any telemetry (trace output, logs) leaves the daemon, it passes through a two-layer redaction pipeline:

Agent output
     │
     ▼
┌──────────────────────────────────────────────┐
│ Layer A — always on                          │
│ • Regex + entropy-based detection            │
│ • API key patterns, high-entropy strings     │
│ → Replaced with <REDACTED:API_KEY:a91f>      │
└─────────────────────┬────────────────────────┘
                      │
                      ▼
┌──────────────────────────────────────────────┐
│ Layer B — optional (requires Presidio)       │
│ • Full PII detection                         │
│ • Names, emails, phones, SSNs, credit cards  │
│ • Prompt injection / jailbreak detection     │
└─────────────────────┬────────────────────────┘
                      │
                      ▼
              Redacted telemetry → cloud

Redaction tokens are salted per-run so they cannot be correlated across runs. Layer A is enabled by default. Layer B requires the presidio-analyzer package and is opt-in.

RBAC Roles

Synapse uses four roles in descending privilege order: owner → admin → operator → viewer. Roles are assigned per-org; a user can have different roles in different orgs.

RoleCapabilities
owner Full control: billing, all settings, all actions, transfer/delete org
admin Deploy/edit agents, manage members, approve HITL, view (not create) secrets
operator Deploy/edit agents, approve HITL gates — cannot see secret values
viewer Read-only: runs, traces, alerts, dashboards — no write actions
Role-based access is enforced at both the API layer (cloud) and the daemon — the daemon validates the requester's role before executing gated actions like approving HITL gates.

Organizations & Teams

An org is the top-level workspace in Synapse. Every resource — daemons, agents, runs, secrets — belongs to an org.

  • Users can belong to multiple orgs and switch between them at any time
  • Each org has its own members, billing subscription, and settings
  • Teams are hierarchical sub-groups within an org, used for organizing members into business units or project groups
Teams are organizational — they do not grant access. Access is governed by the member's role in the org.

Capabilities (MCP & Plugins)

Capabilities extend what agents can do beyond their base model or command. They are installed on daemons and then attached to specific agents.

MCP servers API model agents

Model Context Protocol (MCP) servers expose tools that API model agents can call during a run — file system access, web search, database queries, and more.

Plugins All agent types

Plugins are capability packs that bundle related tools and configuration:

  • Browser automation (Playwright, Puppeteer)
  • GitHub integration (read repos, open PRs, comment on issues)
  • Postgres access (query and mutate databases)
  • Slack integration (send messages, read channels)
An agent can only use capabilities that are both installed on its daemon and attached to it. Installing a capability on a daemon does not grant all agents access — each agent must explicitly opt in.

Capability installation and attachment is governed by guardrails — the capability gating rule type can restrict which capabilities an agent may use, providing an additional layer of least-privilege enforcement.
