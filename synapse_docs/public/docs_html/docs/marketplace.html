Marketplace

Marketplace & Extensions

A curated collection of ready-to-use agent templates, skills, and plugins. Browse in Web UI → Marketplace, click Install — everything lands locally on your daemon. The cloud relays the manifest; your daemon does the installing.

Overview

The Synapse Marketplace is organized into three categories:

  • Agent Templates — pre-built agent configurations with prompts, rulesets, and triggers that you can deploy and customize.
  • Skills — reusable tool bundles that extend what an existing agent can do without rewriting its config.
  • Plugins — system integrations that give agents access to new capabilities: browsers, databases, external APIs.

All three types follow the same trust model: the cloud stores and serves the manifest (name, version, commit hash, permissions required) but never executes anything. Your daemon fetches, verifies, and installs the package locally. The install is pinned to the exact commit hash recorded in the manifest so you always know precisely what code is running.

ℹ

Marketplace items can be installed via the Web UI or the synapse plugin install CLI command. Both paths route through the daemon — nothing is installed cloud-side.

Agent Templates

Agent templates are complete, deployable agent configurations. Each template ships with a default system prompt, a ruleset, recommended triggers, and a declared list of required capabilities. You can deploy a template as-is or use it as a starting point and override any field.

pr-reviewer

Reviews GitHub pull requests, posts inline comments on changed lines, summarizes the diff, and flags potential issues. Requires HITL approval before posting any merge suggestion.

github network HITL: merge verified

support-triage

Classifies incoming support tickets by severity and category, routes them to the appropriate queue, and drafts initial responses for agent review.

network filesystem HITL: send verified

release-captain

Orchestrates the full release process: runs test suites, generates a structured changelog from commit history, and creates a GitHub release draft for human review.

github exec HITL: publish verified

incident-commander

Monitors alert feeds, pages on-call engineers via PagerDuty or Slack, and coordinates incident response. Requires HITL approval before executing any remediation action.

slack network HITL: remediate verified

Installing a template

Templates are installed the same way as any other marketplace item. Once installed, the template appears under Web UI → Agents → New Agent → From Template. You can override any field before deploying.

terminal
$ synapse plugin install template:pr-reviewer
Fetching manifest for template:pr-reviewer@1.4.2...
Verified commit hash a3f8c21d
Installed template:pr-reviewer — ready in Web UI → Agents → New Agent

Skills

Skills are reusable tool bundles. Install a skill and any agent you configure can reference it by name in its tools list. Skills do not carry their own prompts or triggers — they extend an agent's toolbox.

security-scan

Runs SAST analysis and dependency audits on a codebase. Supports Node.js, Python, Go, and Rust projects. Outputs a structured findings report.

filesystem exec verified

sql-explain

Connects to a PostgreSQL or MySQL instance, runs EXPLAIN ANALYZE on slow queries, and returns a human-readable optimization recommendation.

postgres verified

doc-writer

Generates structured API documentation from OpenAPI specs or annotated source code. Outputs Markdown, MDX, or HTML.

filesystem verified

test-author

Generates unit and integration tests from existing source code. Detects the test framework in use and follows existing conventions in the repository.

filesystem exec verified

Installing a skill

synapse plugin install skill:security-scan
synapse plugin install skill:sql-explain
synapse plugin install skill:doc-writer
synapse plugin install skill:test-author

After installation, reference a skill in your agent config under tools:

# synapse-agent.toml
[agent]
name = "code-auditor"
tools = ["skill:security-scan", "skill:doc-writer"]

Plugins

Plugins are system integrations that give agents access to new runtime capabilities. Unlike skills (which are pure tool bundles), plugins may require daemon-level capability grants and environment variables.

⚠

Plugins run at daemon trust level. A plugin has the same process permissions as the daemon itself. Review what a plugin requests before approving the capability grant. No plugin can bypass rulesets or HITL checkpoints, but it does run in your daemon's process space.

Plugin Description Required Capability Required Env Var
browser-use Headless browser control via Playwright. Lets agents navigate pages, fill forms, extract content, and take screenshots. browser —
github Full GitHub API access: read and write PRs, issues, releases, and repo contents. network GITHUB_TOKEN
postgres Direct PostgreSQL connection for query execution and schema introspection. Supports read-only and read-write modes. network DATABASE_URL
slack Post messages, read channels, react to threads, and listen for mentions as a Slack bot. network SLACK_BOT_TOKEN

Installing a plugin

terminal
$ synapse plugin install github
Fetching manifest for plugin:github@2.1.0...
Verified commit hash 7bd9e14f
Capability required: network — approve in Web UI or run with --approve-caps
$ synapse plugin install github --approve-caps
Capability granted: network → agent:* (can be narrowed per-agent)
Plugin installed. Set GITHUB_TOKEN in your daemon environment to activate.

Installing from the Marketplace

You can install any marketplace item via the Web UI or the CLI. Both paths are equivalent — the Web UI generates and dispatches the same install command that the CLI runs directly.

Web UI installation flow

Browse or search

Open Web UI → Marketplace. Use the search bar or filter by category (Templates, Skills, Plugins). Each card shows a description, required capabilities, and the verified badge if applicable.

Click Install

Click the Install button on any marketplace card. This triggers the capability provisioning dialog before anything is downloaded.

Review the capability provisioning dialog

The dialog lists every permission the plugin or skill will need: capabilities (browser, network, filesystem, exec) and any required environment variables. Read this carefully — you are granting daemon-level access.

Confirm

Click Confirm. The cloud sends the install command to your daemon over the existing outbound WebSocket connection. Your daemon fetches the package, verifies the commit hash, and installs it locally. No files are staged cloud-side.

Agent picks up the new capability

The next time an agent runs, the new tool or plugin is available. Running agents pick up the change on their next invocation — there is no restart required for the daemon.

CLI installation

# Install a plugin
synapse plugin install github

# Install a skill
synapse plugin install skill:security-scan

# Install a template
synapse plugin install template:pr-reviewer

# Install a specific version
synapse plugin install github@2.1.0

# Skip the interactive capability approval prompt
synapse plugin install browser-use --approve-caps

Capability Provisioning Flow

When a plugin requires a system capability — browser, network, filesystem, or exec — Synapse shows a provisioning dialog before installation proceeds. Grants are per-agent by default, which means one agent's browser permission does not automatically extend to other agents running in the same daemon.

How it works

  • The capability dialog shows the exact permission name, what the plugin will use it for, and which agents it will apply to.
  • An admin (any user with the admin role on the daemon) must approve the grant.
  • Grants are stored in the daemon's local config and reflected in the Web UI → Daemon → Capabilities panel.
  • Grants can be revoked at any time without uninstalling the plugin — the plugin simply stops being able to exercise the capability.

Capability grant table

Agent Capability Granted By Granted At Scope
pr-reviewer network admin@example.com 2025-11-14 09:22 UTC github.com only
incident-commander network admin@example.com 2025-11-14 09:25 UTC unrestricted
ui-tester browser ops@example.com 2025-11-20 14:03 UTC internal domains
release-captain exec admin@example.com 2025-11-21 11:48 UTC make, npm, cargo
✓

Use scoped grants wherever possible. Granting network to a specific domain rather than unrestricted limits blast radius if an agent is ever misused or misconfigured.

Publishing to the Marketplace

Anyone can submit a plugin, skill, or template to the Synapse Marketplace. Community-submitted items receive a community badge; items reviewed and validated by the Synapse team receive the verified badge.

Step 1 — Create the manifest

Add a synapse-plugin.toml file to the root of your repository:

[plugin]
name        = "my-plugin"
version     = "1.0.0"
type        = "plugin"           # plugin | skill | template
description = "A short description shown in the marketplace."
author      = "Your Name <you@example.com>"
license     = "MIT"

[capabilities]
required = ["network"]           # capabilities the plugin needs

[env]
required = ["MY_API_KEY"]        # env vars the user must provide
optional = ["MY_API_ENDPOINT"]   # env vars with defaults

[compat]
min_daemon = "0.9.0"

Step 2 — Push to a public git repository

The manifest and all plugin source must be in a publicly accessible git repository. Synapse pins installs to the exact commit hash, so all published versions must remain immutable — do not force-push tags.

Step 3 — Submit via Web UI

Go to Web UI → Marketplace → Submit. Paste your repository URL and select the version tag to submit. The submission form validates your manifest against the current schema before sending.

Review process

  • Community review — automated checks: manifest schema validation, no hardcoded secrets, declared capabilities match actual API calls. Typically completes within minutes. Items pass as community.
  • Verified review — manual code review by the Synapse team. Checks for security issues, capability over-claiming, and adherence to the plugin API contract. Submit a verified review request via the marketplace submission form. Typically takes 3–5 business days.
ℹ

To update a published plugin, push a new tag to your repository and re-submit the new version via Web UI → Marketplace → Submit. Old versions remain installable by pinning: synapse plugin install my-plugin@1.0.0.

Security Model

Understanding how plugins interact with the Synapse security model is essential before granting capabilities.

Trust level

Plugins run inside the daemon process. They share the daemon's OS-level permissions and can access anything the daemon can access — subject to the capability grants you have approved. There is no sandbox isolation between the daemon and its plugins. Treat a plugin install with the same scrutiny you would apply to installing a system package.

What plugins cannot do

  • A plugin cannot bypass a ruleset. All agent actions, including tool calls made via plugins, pass through the ruleset evaluation engine before execution.
  • A plugin cannot skip HITL checkpoints. If an agent's ruleset requires human approval before a specific action, the plugin must wait for that approval even if the plugin code itself does not reference HITL.
  • A plugin cannot exfiltrate raw secrets to the cloud. The cloud connection from the daemon is write-only for results and read-only for commands — plugins cannot open new outbound channels.

Commit-hash pinning

Every installed plugin is pinned to the exact commit hash recorded in the manifest at install time. Synapse will not silently update a plugin. To update, you must explicitly run synapse plugin update <name>, which re-runs the capability provisioning dialog so you can review what changed.

Auditing installed plugins

Run synapse daemon status --verbose to see all installed plugins, their pinned commit hashes, the capabilities they hold, and the last time each was used by an agent. This output is also visible in Web UI → Daemon → Plugins.

⚠

If a plugin's commit hash no longer matches the repository history (e.g. after a force-push), Synapse will flag it as tampered and refuse to load it until you explicitly re-install or remove it. Never force-push a published plugin tag.
