Unit 9

Scheduling & Webhooks

Trigger agents on a schedule, respond to external events via inbound webhooks, and push Synapse events to your own endpoints. Everything runs through the daemon — the cloud brokers but never executes.

Cron Scheduling

Synapse supports standard five-field cron syntax for recurring agent runs. Schedules are stored in the cloud and dispatched to whichever daemon is online for the org at fire time.

Cron syntax

The five fields map to: minute hour day-of-month month day-of-week. Use * for "any", */n for "every n", and comma-separated values for lists.

Expression Meaning Typical use case
0 * * * * Every hour, on the hour Hourly data sync, rate-limited polling
0 2 * * * Every day at 2:00 AM Nightly reports, database backups, digest emails
0 9 * * 1-5 Weekdays (Mon–Fri) at 9:00 AM Morning standup summaries, business-hours alerts
*/15 * * * * Every 15 minutes Health checks, queue consumers
0 0 1 * * First day of each month at midnight Monthly billing reports, archive jobs

Timezone support

Schedules default to UTC. You can set a per-schedule timezone using IANA timezone identifiers (e.g. America/New_York, Europe/London, Asia/Tokyo).

shell
$ synapse agent schedule add standup-bot \
  --cron "0 9 * * 1-5" \
  --timezone "America/New_York"
Schedule created: standup-bot (cron: 0 9 * * 1-5, tz: America/New_York)

Configure via Web UI

Navigate to Agent → Triggers tab → Add Schedule. Enter the cron expression, select a timezone from the dropdown, and click Save. The schedule activates immediately once a daemon for your org comes online.

Interval Triggers

For simpler recurring patterns — every N minutes or hours — use interval triggers instead of cron. The daemon fires the agent roughly at the specified interval, measured from the last run completion (not wall-clock alignment).

Specifying an interval

Set interval in the agent config or via the CLI using a duration string:

  • interval: "30m" — every 30 minutes
  • interval: "4h" — every 4 hours
  • interval: "1h30m" — every 90 minutes
shell
$ synapse agent schedule add health-checker --interval 30m
Schedule created: health-checker (interval: 30m)
 
$ synapse agent schedule add metrics-poll --interval 4h
Schedule created: metrics-poll (interval: 4h)
💡

Use interval triggers for polling agents and heartbeat checks. Use cron when runs must align to wall-clock time (e.g. "send a report every Monday at 9am").

Overlap Policies

If the daemon fires a schedule but the agent from the previous run is still executing, the overlap policy decides what to do. Set overlap in the agent config or via the Triggers tab.

Policy Behavior When to use
coalesce (default) If a run is already active, skip the scheduled fire. The next scheduled tick will try again normally. Most agents — prevents pile-ups when a run takes longer than the interval. Safe for idempotent jobs.
run_once Enforces a global limit of one live instance at any moment. Any additional trigger (scheduled or manual) is queued and starts only after the current run finishes. Agents that mutate shared state and must not run concurrently — e.g. a schema migration agent or a ledger reconciler.
allow Always start a new run regardless of how many instances are already running. No deduplication. Stateless fan-out agents, parallel enrichment pipelines, or any workload explicitly designed for concurrent execution.
⚠

allow policy warning: with short intervals and slow agents, allow can spawn hundreds of concurrent runs. Monitor run counts in the dashboard or set a concurrency budget with max_concurrency.

Manual Triggers

Any agent can be triggered on demand — from the Web UI, the CLI, or via an inbound webhook. Manual runs respect the same overlap policy as scheduled runs.

Web UI

Open the agent detail page and click Run now on the Runs tab. A dialog lets you override environment variables for this specific run before confirming.

CLI

Use synapse agent run <name> to trigger a run from the terminal. Pass --env KEY=val one or more times to inject one-off environment variable overrides that are scoped to this single run.

shell
# Trigger with a one-off env override
$ synapse agent run pr-reviewer --env REPO=myorg/myapp
Queued run for pr-reviewer
Run ID: run_01hw9yk3...
Run started on daemon daemon-prod-1
 
# Follow logs in real-time
$ synapse run logs run_01hw9yk3 --follow
ℹ

Webhook-triggered runs are classified as manual for RBAC purposes. They are subject to the same role permissions as "Run now" clicks in the Web UI — not to scheduled-trigger permissions.

Inbound Webhooks

Each agent can be assigned one or more unique inbound webhook URLs. Any HTTP POST to that URL triggers an agent run. The request body is made available inside the run as the WEBHOOK_PAYLOAD environment variable (raw string, typically JSON).

HMAC-SHA256 signing

Every webhook URL includes an HMAC-SHA256 signature in the query string. Synapse validates the signature before dispatching the run. This prevents unauthorized callers from triggering your agents. The signing secret is generated by Synapse and never leaves your account.

ℹ

Rotating your HMAC secret: if a webhook URL is compromised, go to Agent → Triggers → [webhook] → Rotate Secret. The old URL is immediately invalidated. Pending in-flight requests signed with the old secret will be rejected. Update any upstream callers with the new URL before rotating in production.

Setup steps

Open the Triggers tab

Navigate to your agent in the Web UI. Click the Triggers tab on the agent detail page.

Add a webhook

Click Add Webhook. Synapse generates a unique signed URL immediately. You can optionally label the webhook (e.g. "GitHub PR events").

Copy the URL

Click the copy icon next to the URL. Paste it into your upstream service (GitHub, PagerDuty, Stripe, etc.) as the webhook destination. Configure the upstream to send POST requests.

Test the endpoint

Use the Send test payload button to fire a synthetic POST. Check the Runs tab to confirm the agent started and received WEBHOOK_PAYLOAD.

Manage or revoke

Rotate the signing secret or delete the webhook entirely from the same Triggers panel. Deletion is immediate and permanent.

Accessing the payload

Inside your agent process, read WEBHOOK_PAYLOAD from the environment:

import json, os

payload = json.loads(os.environ.get("WEBHOOK_PAYLOAD", "{}"))
print("Received event:", payload.get("action"))

Outbound Webhooks

Synapse can POST lifecycle events from your agents and organization to HTTP endpoints you control. Use this to integrate with monitoring systems, chat platforms, ticketing tools, or custom automation pipelines.

Setup

Go to Settings → Webhooks → Add Endpoint. Enter your HTTPS URL and choose which event types to subscribe to. You can add multiple endpoints, each with its own event filter.

Event payload format

All outbound events are delivered as JSON POST bodies with a consistent envelope:

{
  "event":     "run.completed",
  "org_id":    "org_01hw...",
  "agent_id":  "agent_01jx...",
  "run_id":    "run_01hw9yk3...",
  "timestamp": "2026-06-10T14:32:07Z",
  "data": {
    "status":        "success",
    "duration_ms":   4821,
    "exit_code":     0
  }
}

The data field varies by event type — see the Event Types table below for the key fields each event carries.

Request headers

Each outbound request includes a X-Synapse-Signature header containing an HMAC-SHA256 signature of the raw request body, signed with your endpoint's secret. Verify this header before processing events.

Event Types

All events are available for outbound webhook subscriptions and notification routing. Subscribe selectively to keep signal-to-noise high.

Event Description Key data fields
run.started An agent run has been accepted and started on the daemon. trigger, daemon_id
run.completed A run finished successfully (exit code 0). status, duration_ms, exit_code
run.failed A run exited with a non-zero code or was killed. exit_code, error_message, duration_ms
approval.requested An agent hit a human-in-the-loop checkpoint and is awaiting approval. checkpoint_id, prompt, timeout_at
approval.resolved An approval checkpoint was approved or denied. checkpoint_id, decision, resolved_by
alert.fired An agent-defined alert condition was triggered (e.g. error rate threshold). alert_name, severity, value, threshold
budget.exceeded An agent or org has exceeded a configured cost or token budget. budget_type, limit, actual, period
daemon.offline A daemon lost its heartbeat connection to the cloud broker. daemon_id, last_seen_at

Notification Routing

Notification channels let you route events to Slack, Discord, or Email without writing custom webhook handlers. Configure channels in Settings → Notifications, then define rules that map event types and severity levels to those channels.

Channel setup

Go to Settings → Notifications → Add Channel. Supported channel types:

  • Slack — paste a Slack incoming webhook URL and optionally pin to a channel.
  • Discord — paste a Discord webhook URL from Server Settings → Integrations.
  • Email — enter one or more recipient addresses; Synapse sends via its transactional mail provider.

Routing rules

Rules are evaluated top-to-bottom. Each rule specifies the event types it matches, a minimum severity level, and the target channel. Example configuration (shown in YAML-like notation for readability):

notification_rules:
  - name: "Critical failures to Slack ops"
    events: [run.failed, daemon.offline, budget.exceeded]
    severity: critical
    channel: slack-ops-alerts

  - name: "All run events to email digest"
    events: [run.started, run.completed, run.failed]
    severity: info
    channel: email-team-digest

  - name: "Approvals to Discord"
    events: [approval.requested, approval.resolved]
    severity: info
    channel: discord-approvals

  - name: "Budget warnings to Slack"
    events: [budget.exceeded, alert.fired]
    severity: warning
    channel: slack-cost-alerts
💡

Keep critical-severity rules at the top. If multiple rules match an event, all matching channels receive the notification — there is no short-circuit.

Delivery Reliability

Synapse retries failed outbound webhook deliveries automatically. A delivery is considered failed if the endpoint returns a non-2xx HTTP status code or does not respond within 10 seconds.

Retry schedule

Attempt Delay after previous attempt Cumulative time elapsed
1 (initial) — 0s
2 30 seconds ~30s
3 2 minutes ~2m 30s
4 10 minutes ~12m 30s
5 (final) 30 minutes ~42m 30s

After five failed attempts the event is marked dead. No further retries are made. You can manually replay dead events from the delivery log.

Delivery log

Every delivery attempt — successful or failed — is recorded. To inspect it: go to Settings → Webhooks → [your endpoint] → Delivery Log. Each entry shows the event type, HTTP status code returned, response time, and retry count.

ℹ

Delivery logs are retained for 30 days. Use the Replay button on any entry to re-send a specific event to your endpoint, regardless of its original delivery status.

Idempotency

Because events can be delivered more than once (on retries), your endpoint should handle duplicate deliveries gracefully. Use the run_id or checkpoint_id from the event payload as an idempotency key when writing to databases or triggering downstream actions.
