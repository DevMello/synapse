App Shell & Navigation

Every screen in Synapse shares the same three-region layout: a sticky top header, a fixed left sidebar, and a grid-textured main content area. The shell is rendered by the root AppLayout component and wraps each screen in a React Suspense boundary so lazy-loaded tabs load without a full-page flash.

Sidebar (264 px)

The sidebar is always visible on desktop and collapses to a slide-out drawer on smaller viewports.

  • Logo & brand — clicking the Synapse logo navigates to the Dashboard.
  • Connect a device — orange primary button that opens the Connect flow for adding a new daemon.
  • Monitoring — Dashboard, Runs, Approvals (with live badge count), Alerts.
  • Agents & Daemons — Agents, Daemons.
  • Marketplace — browse and install agent templates, skills, and plugins.
  • Settings — Webhooks, Notifications, Settings.
  • Workspace footer — current org name with plan badge, avatar, and a "Switch org" shortcut.

Nav items highlight with an orange left-border accent when active. Badge counts on Approvals and Alerts update in real-time via Supabase Realtime.

Header (sticky 64 px)

  • Breadcrumb path — reflects the current route hierarchy (e.g. Agents › my-agent › Editor). Each segment is a clickable link except the last.
  • Search bar — click or press Cmd+K / Ctrl+K to open the command palette. The palette provides fuzzy search across agents, daemons, and recent runs, plus quick-action shortcuts (new agent, open connect, jump to approvals).
  • Notification bell — shows an unread-count badge; clicking opens the Notifications panel.
  • Alerts icon — shows a count badge for unresolved anomaly alerts.
  • User avatar — opens a dropdown with links to Account Security, Organizations, and Sign out.

Content area

The main content area sits to the right of the sidebar below the header. It uses a subtle grid-dot texture on the warm bone background (#f4efe8). Each screen manages its own max-width and padding; most cap at 1 280 px with 32 px horizontal padding.

Dashboard /

The dashboard gives you a fleet-wide overview at a glance. All data refreshes live via Supabase Realtime — run status changes and new approvals appear without a manual page refresh.

Metric cards (4)

CardWhat it shows
Daemons online Online count vs. total registered (e.g. 3 / 5 online). Clicking navigates to the Daemons screen.
Active runs Number of runs currently in running state across all agents.
Spend today Cumulative USD spend since midnight UTC. Includes model tokens, plugin usage, and any metered capabilities.
Open approvals Runs paused at a HITL gate awaiting a human decision. Clicking navigates to the Approvals screen.

Live active runs table

A real-time table of every run currently executing. Columns: Agent (name + link to agent detail), Trigger (icon for schedule / webhook / manual / agent), Daemon (host daemon name), Started (relative time), Cost (running USD total), Status chip. Click any row to navigate directly to the run's trace view.

Top agents by spend

Four agent cards ranked by descending daily spend. Each card shows: engine icon, agent name, engine identifier, daily spend (USD), and a miniature sparkline chart plotting recent run costs. Useful for spotting runaway spend before it becomes a problem.

Daemon mini-list

One row per connected daemon. Each row shows the daemon hostname, OS icon, a status dot (green = online, gray = offline), a 12-beat heartbeat strip visualising the last 12 heartbeat results, and a "last seen" timestamp.

Alert mini-feed

The latest three anomaly alerts with severity chip and timestamp. Click "View all" to go to the full Alerts screen.

The dashboard subscribes to Supabase Realtime channels for runs, approvals, and daemon_heartbeats. All widgets update automatically — no polling interval to configure.

Agents /agents

The Agents screen lists every agent registered to your org. You can browse, filter, search, and launch the New Agent wizard from here.

List vs. grid view

Toggle between a 3-column card grid and a compact list view using the view-toggle buttons in the top-right corner. The preference is stored locally and remembered between sessions.

Filters

  • Status — All / Running / Unavailable
  • Search — fuzzy match on agent name, engine identifier, or host daemon name

Agent card

Each card in grid view shows:

  • Engine icon (model avatar)
  • Agent name and type chip (API model or CLI tool)
  • Engine name (e.g. claude-sonnet-4-5)
  • Host daemon name
  • Daily spend (USD)
  • Status chip — running, passed, blocked, or idle

Clicking a card opens the Agent Detail screen.

New Agent Wizard

Click New Agent to open the 5-step modal wizard:

  1. Name — agent name and optional description
  2. Host — select the daemon that will run this agent
  3. Engine — choose API model (select provider + model) or CLI tool
  4. Template — optionally pre-populate from a Marketplace template
  5. Deploy — review settings and deploy

See the Agent Management docs for the full wizard walkthrough.

Agent Detail — Overview Tab /agents/:id

Clicking any agent opens its detail page. The page is composed of a persistent hero row at the top followed by a 12-tab panel.

Hero row

  • Engine avatar — large icon representing the model or tool type
  • Agent name — rendered in the display heading style
  • Status chip — running animates with a pulsing orange dot; idle is gray
  • Enabled / Disabled toggle — disabling prevents new runs from starting (ongoing runs are not interrupted)
  • Run now button — triggers a manual run immediately

Agent Detail — 12 Tabs

The tab bar below the hero row gives access to every facet of the agent. Tabs are lazy-loaded: heavy ones (Editor, Analytics) ship in their own JS chunk and render behind a Suspense boundary.

Editor

A full CodeMirror 6 Markdown editor for the agent's system prompt. Saving the editor creates an immutable version snapshot (visible in the Versions tab). A diff view toggles between the current working copy and any previously selected version, rendering additions in green and deletions in red.

Versions

A table of all snapshot versions of the system prompt. Columns: Label, Author, Timestamp, Message, Tags (e.g. production known-good). Each row has a Set as active button that rolls back the live prompt to that snapshot without requiring a re-deploy.

Schedule

Configure when the agent runs automatically:

  • Cron expression — standard 5-field cron (e.g. 0 9 * * 1-5 for 09:00 on weekdays)
  • Interval — simple repeat interval (e.g. every 15 minutes)
  • Next run time — previewed in your local timezone based on the current expression
  • Overlap policy — Coalesce (skip extra triggers while one run is in flight), Skip, or Run once (queue at most one pending run)

Tools & MCP

Lists every capability (MCP server or plugin) installed on the host daemon. Each row shows the capability name, kind chip, and an on/off toggle controlling whether this agent can use it. Capabilities toggled off for this agent are never exposed at runtime even if installed on the daemon. A Request capability button initiates provisioning for capabilities not yet installed.

Plugins

Ecosystem plugins that extend the agent with domain-specific tools:

  • browser-use — Playwright-backed web automation
  • github — read repos, open PRs, comment on issues
  • postgres — query and mutate databases
  • slack — send and read messages

Each plugin shows its install status and exposes a per-agent settings form (e.g. target Postgres connection string, Slack workspace).

Environment

Manage the env vars injected into this agent's runtime. Columns: Key, Type (secret / plain), Origin (cloud = stored in Supabase Vault; local = stored only on the daemon), Last updated, Updated by.

Secret values are write-only. Once saved, the value is never displayed in the UI again — only the key name and metadata are shown. Overwrite or delete to change a secret.

Memory

Per-agent key-value memory store. The table shows: Key, Namespace, Tags, Size (bytes), Last updated. Actions: search by key or tag, view value inline, edit, delete. The Pre-load snapshot action lets you seed a batch of memory entries from a JSON file before starting a run — useful for bootstrapping context.

Runs

Agent-scoped run history (same columns as the global Runs screen). Clicking a row expands an in-line trace viewer below the table showing the full step-by-step execution log for that run — no navigation away from the page needed.

Orchestration

Manage which sub-agents this agent is authorized to spawn. Each grant row shows:

  • Grantee agent — the sub-agent that may be spawned
  • Verbs — permitted actions: run / create / edit
  • Max depth — how many levels of spawning are allowed
  • Max fan-out — concurrent sub-agents cap
  • Tree budget (USD) — cumulative spend limit for the entire sub-tree
  • Expiry — optional grant expiration time

Mint new grant opens a form to add a new orchestration grant. Revoke on an existing row invalidates it immediately.

Orchestration grants are signed with ed25519 and verified by the daemon before a sub-agent run is started. A grant cannot be self-issued — an operator or admin must create it.

Logs

Structured log viewer for the most recent run (or any selected run). Logs stream in real time during an active run. Filter by TAG: PLAN BUILD QA MCP. Each row shows: timestamp, TAG, severity indicator, and message rendered in JetBrains Mono.

Analytics

Four Recharts visualisations (lazy-loaded, async chunk):

ChartTypeDescription
Daily spendBar chartUSD cost per day over the selected date range
Token usageStacked barInput tokens vs. output tokens per day
Run latencyLine chartMedian and p95 run duration (seconds) over time
Tool call distributionPie chartRelative share of each tool/MCP call type

Use the date-range picker in the top-right of the Analytics tab to zoom into a specific window.

Daemons /daemons

The Daemons screen shows every device that has been authorised to run agents on behalf of your org. Each daemon is represented by an expandable card.

Daemon card

The collapsed card header shows:

  • Hostname — the machine's hostname as reported by the daemon
  • OS icon — monitor (Windows), apple / smartphone (macOS), server (Linux)
  • Status dot — green online, gray offline
  • Uptime % — percentage of heartbeats received in the past 24 h
  • CPU % — last reported CPU utilisation
  • Memory % — last reported RAM utilisation
  • Active run count — runs currently executing on this daemon
  • Last seen — relative time of the most recent heartbeat
  • Daemon version — synapse-worker semver
  • Tags — user-defined labels for grouping (e.g. prod, eu-west)
  • 12-beat heartbeat strip — coloured squares for the last 12 heartbeat intervals (green = received, red = missed)

Expanding the card reveals two additional sections:

Agents list

Each agent hosted on this daemon appears as a row with name and current status chip. Click an agent name to navigate to its detail page.

Capabilities grid

Every capability available to (or installed on) this daemon. Columns: Name, Kind (MCP server / plugin), State chip, Built-in badge. An Install button appears for capabilities that are available but not yet provisioned.

Revoke device

The Revoke device button at the bottom of the expanded card permanently removes the daemon's authorisation. The daemon will no longer be able to connect to the cloud control plane. Any agents hosted on the daemon will be marked unavailable. This action requires admin role and shows a confirmation dialog before proceeding.

Revoking a daemon immediately terminates all ongoing runs on that machine. This cannot be undone — you will need to re-run synapse login on the device and go through the Connect flow again to re-authorise it.

Connect (Device Authorization) /connect

The Connect flow authorises a new daemon (device) to join your org. It follows a 3-step device-authorization pattern: the daemon displays an 8-character code, and you confirm it in the Web UI.

Step 1 — Enter code

Enter the 8-character alphanumeric code that appeared in your terminal after running synapse login. The code input is split into two 4-character segments for readability (e.g. ABCD · 1234). A Paste from clipboard button auto-fills both segments.

Step 2 — Verify device

Before approving, Synapse shows you the metadata of the device requesting access:

  • Hostname — machine name reported by the daemon
  • OS / Platform — operating system and architecture
  • IP address + geo — approximate city/country derived from the request IP
  • Code requested at — timestamp when the daemon generated the code
Always verify the IP address and hostname match the machine you intended to connect. Deny if you did not start this from your own terminal.

Step 3 — Approved

After clicking Approve, the daemon receives a signed token and completes its registration. The success screen shows the new daemon name and a link to view it in the Daemons screen.

Runs /runs

The Runs screen is the fleet-wide run history — every execution across every agent in your org, newest first.

Metric strip

Four summary cards at the top of the page:

  • Active runs — runs currently in running state
  • Runs today — count of runs started since midnight UTC
  • Spend today — cumulative USD cost for today's runs
  • Tokens today — total tokens consumed today (input + output)

Approvals /approvals

The Approvals screen is the live Human-in-the-Loop (HITL) queue. When an agent hits a HITL gate in its guardrail policy, the run pauses and a card appears here. The sidebar badge count updates in real time. See the HITL Gates docs for how to configure gates.

Pending section

Each pending approval is a card showing:

  • Agent name and run ID
  • Gate label — the name of the HITL gate that triggered the pause
  • Payload preview — a truncated view of what the agent was about to do
  • Waiting since — how long the run has been paused
  • Reason textarea (optional) — annotate your decision for the audit log
  • Approve and Deny action buttons

Approving resumes the run from where it paused. Denying terminates the run with exit code 2 (denied).

Resolved section

The most recently resolved approvals are shown below the pending queue. Columns: Agent, Decision (chip), Decided by, When, Reason. This section is read-only and provides an audit trail.

Only users with operator, admin, or owner roles can approve or deny HITL gates. Viewers see the queue but the action buttons are disabled.

Marketplace /marketplace

Browse and install curated agent templates, skills, and plugins from the Synapse Marketplace.

Kind filter tabs

Three tabs narrow the catalogue to a specific kind:

  • Agents — pre-configured agent templates ready to deploy
  • Skills — reusable system-prompt fragments that add a specific capability to any agent
  • Plugins — installable daemon capabilities (tools, MCP servers)

Filter the visible items by name or description. Results update as you type.

Webhooks /webhooks

Webhooks let external services trigger agent runs by POSTing to an inbound URL. Each webhook is tied to a target agent.

Webhook list

ColumnDescription
NameHuman-readable label
Endpoint URLThe HTTPS URL to POST to, with a copy button
SecretHMAC secret used to sign and verify payloads — reveal/hide toggle plus copy button
Target agentThe agent that runs when this webhook fires (link to Agent Detail)
DeliveriesCount of POST requests received
CreatedCreation date
EnabledToggle to pause/resume the webhook without deleting it

Create webhook

Click New Webhook and provide: Name and Target agent (select from dropdown). A URL and secret are generated automatically.

Delivery history

Expand any webhook row to see a log of every delivery attempt. Columns: Timestamp, Webhook name, HTTP status code, Response snippet, Run status. Failed deliveries (non-2xx) are highlighted in red.

Synapse signs every inbound webhook payload with HMAC-SHA256. The signature is sent in the X-Synapse-Signature header. Your service should verify this signature against the stored secret before trusting the payload. See the API Reference for the verification algorithm.

Notifications /notifications

Configure where Synapse sends event notifications — Slack, Discord, or email — and create routing rules that map event types to specific channels.

Channels tab

Add and manage notification destinations:

  • Slack — provide a channel name and bot token. The bot must have chat:write scope in the target workspace.
  • Discord — provide a guild ID and channel ID. Use a bot token with SEND_MESSAGES permission.
  • Email — add an email address. Synapse sends HTML email via the platform mailer.

Each channel has an enabled/disabled toggle so you can pause delivery without deleting the config.

Routing rules tab

A routing rule maps one event type to one destination channel, optionally scoped to specific agents.

FieldOptions
Event type hitl.approval — HITL gate paused
alert.prompt-injection — injection attempt detected
run.failed — run exited non-zero
run.passed — run exited cleanly
budget.exceeded — spend exceeded configured limit
Agent scope All agents / specific agent (by name) / agents matching a tag
Destination Select from configured channels in the Channels tab

Multiple rules can match the same event — every matching rule fires, so a single run.failed event can notify both Slack and email if two rules are configured.

Settings /settings

The Settings screen manages org-level configuration across six tabs.

Profile tab

View and edit core org metadata:

  • Org name — display name used throughout the UI
  • Region — data residency region (set at org creation, read-only after)
  • Plan — current subscription tier (Free / Pro / Team / Enterprise)
  • Owner — current owner's email
  • Created — org creation date
  • Workspace ID — immutable UUID used in API calls

Members & RBAC tab

Current member table columns: Name, Email, Role (owner / admin / operator / viewer), MFA status, Last active.

Invite a new member by entering their email and selecting a role, then clicking Send invite. Pending invitations are listed below the members table with a Revoke invite button.

RolePermissions
owner Full control: billing, all settings, all actions, transfer or delete org
admin Deploy and manage agents, manage members, approve HITL, view (not create) secrets
operator Deploy and edit agents, approve HITL gates — cannot see secret values
viewer Read-only: runs, traces, alerts, dashboards — no write actions

Teams tab

Teams are hierarchical sub-groups within an org used for organising members into business units or project groups. Teams do not grant access — that is governed by each member's org role.

  • Team tree — hierarchical view of all teams with parent/child nesting
  • Create team — name + optional parent team
  • Per-team member list — add or remove members from a team
  • Delete team — requires confirmation; child teams are promoted to top-level

Billing & Usage tab

  • Current plan — name, seat limit, included compute credits
  • Usage this month — spend (USD) and seat count vs. plan limits, with progress bars
  • Invoice history — table: period, amount, status chip, PDF download link

API Tokens tab

Create bearer tokens for programmatic API access.

  • Create token — name + scope: read (GET only) or deploy (read + write + trigger runs). Token value is shown once on creation and never again.
  • Token list — name, prefix (first 8 chars), scope, created date, last used date
  • Revoke button — requires confirmation; invalidates the token immediately
Copy the token value immediately after creation. It is shown only once and cannot be recovered. If you lose it, revoke and create a new token.

Authentication tab

Org-wide authentication policy (owner-only access):

  • Require MFA for all members — when enabled, members without an enrolled MFA factor are blocked from the UI until they complete enrollment
  • Per-member MFA status — table showing each member and whether they have an enrolled factor

Account Security /account/security

Personal multi-factor authentication enrollment. Each user manages their own MFA factors here, independent of org membership.

Enrolled factors list

Columns: Type (TOTP / WebAuthn), Label (name you gave the factor), Status (verified / unverified), Remove button. You must have at least one remaining verified factor before you can remove another — this prevents accidental lockout.

Add TOTP (authenticator app)

  1. Click Add authenticator app.
  2. Scan the QR code with your authenticator app (Google Authenticator, Authy, 1Password, etc.). The manual key is displayed below the QR code if you cannot scan.
  3. Enter the 6-digit code shown in your app and click Verify.
  4. Give the factor a label (e.g. iPhone — Authy) and save.

Add WebAuthn (security key or passkey)

  1. Click Add security key / passkey.
  2. Your browser prompts for your biometric or hardware key (Touch ID, Face ID, Windows Hello, or FIDO2 hardware key).
  3. Complete the browser prompt.
  4. Give the credential a label (e.g. YubiKey 5C) and save.

Recovery codes

Recovery codes are single-use bypass tokens that work when all MFA factors are inaccessible (e.g. lost phone, broken key). Click View recovery codes to reveal your codes. Click Regenerate to invalidate the existing set and generate a fresh batch.

Store recovery codes somewhere safe — a password manager, printed sheet in a secure location, or offline backup. They are your last resort if you lose access to all your MFA factors. Each code can only be used once.

Organizations /account/organizations

Switch between your workspaces or create a new org. Each org is an isolated workspace with its own daemons, agents, members, and billing.

Personal workspace card

Your personal workspace always appears first. If it is the currently active org it shows an Active badge. The personal workspace cannot be deleted or transferred.

Org cards

Each org you are a member of appears as a card showing:

  • Org name
  • Plan badge — Free / Pro / Team / Enterprise
  • Member count
  • Switch to button — sets this org as the active context; all subsequent API calls are scoped to it
  • Open settings button — navigates directly to this org's Settings page

Create org

Enter a name in the Create a new organization form and click Create. You are automatically set as the owner of the new org.

Additional organizations beyond your personal workspace are available on Team and Enterprise plans. On the Free or Pro plan you can create one org (your personal workspace). Upgrade from Settings → Billing & Usage to add more.
